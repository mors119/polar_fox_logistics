# 아키텍처

## 방향

이 프로젝트는 Google Apps Script 애플리케이션입니다.  
프레임워크 스타일 계층 분리보다, 기능을 바로 찾고 바로 수정할 수 있는 구조를 우선합니다.

핵심 원칙:

- 기능 중심 폴더
- 작은 함수
- 명시적 import
- 얇은 진입점
- 최소한의 공용 모듈

## 현재 구조

```text
src/
├── config/
├── reports/
├── shared/
├── triggers/
├── webapp/
└── index.ts
```

### `config/`

- Script Properties 키 상수만 둡니다.
- 설정 해석 로직은 `shared/config.ts`에서 처리합니다.

### `reports/`

- 일일 리포트 발송
- 리포트 스냅샷 저장
- 스프레드시트 메뉴 액션

리포트 관련 기능과 타입을 한곳에서 찾을 수 있게 둡니다.

### `shared/`

- Sheets 읽기/쓰기
- Gmail 발송
- Drive 파일 생성
- UI 메뉴/알림
- JSON 응답/로깅/공통 타입

여기는 공통 유틸리티 모음이지, 별도 인프라 계층이 아닙니다.

### `triggers/`

- `onOpen`
- `onEdit`
- 시간 기반 트리거

Apps Script 이벤트별 진입 함수를 둡니다.

### `webapp/`

- `doGet`
- `doPost`

웹앱 응답 생성과 payload 저장 로직을 둡니다.

## 의존 방식

```text
index.ts
  -> triggers / webapp / reports
  -> shared
```

복잡한 의존성 역전은 사용하지 않습니다.  
기능 모듈이 필요할 때 `shared` 함수를 직접 가져다 씁니다.

## 테스트 방식

- 테스트는 클래스가 아니라 함수 단위로 작성합니다.
- Apps Script 전역 객체가 필요한 부분은 의존 함수 주입으로 대체합니다.
- 비즈니스 흐름 검증을 우선하고, 래퍼 함수 자체는 얇게 유지합니다.

## 왜 이 구조가 더 적합한가

- Apps Script 개발자는 보통 이벤트 함수와 업무 함수 중심으로 코드를 찾습니다.
- 포트, 어댑터, 컨테이너를 거치지 않아 실행 흐름이 짧습니다.
- 한 기능이 여러 계층으로 흩어지지 않아 수정 속도가 빠릅니다.
- 이 저장소는 재사용 프레임워크가 아니라 운영 자동화 앱이므로, 확장성보다 읽기 쉬움이 더 중요합니다.
